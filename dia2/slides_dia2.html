Ideia central

Não existe “a melhor IA”. Existe a ferramenta que encaixa melhor na tarefa, no risco e no orçamento.

Cinco famílias de uso

1. Chat na web

Pensar, escrever, resumir, analisar arquivos e pesquisar.

2. Copilots / IDEs

Ajudar a escrever código dentro do editor.

3. Especializadas

Pesquisa, documentos, fontes e conhecimento pessoal.

4. Agentes

Executar tarefas em arquivos, terminal, browser e repositórios.

5. APIs

Colocar IA dentro de produtos, planilhas e rotinas.

Perguntas que guiam a escolha

Qual é o output?

Texto, planilha, código, gráfico, decisão ou automação?

Qual é o contexto?

Informação pública, arquivo próprio, código local, dados sensíveis?

Como validar?

Fonte, teste, cálculo, revisão humana, comparação com benchmark?

Custos

Pagar por IA é infraestrutura

  • Para uso profissional, IA paga já se parece com internet, cloud e software básico
  • O plano gratuito serve para experimentar
  • O plano pago remove fricções: limites, modelos melhores, arquivos, contexto e velocidade
  • O custo relevante não é só a mensalidade: é tempo, erro evitado e qualidade da entrega

Mensagem prática

Se a ferramenta poupa algumas horas por mês, US$20 deixa de ser caro. O problema é pagar sem mudar o workflow.

Boas práticas

Prompt bom tem estrutura

Inclua

  • papel da IA
  • objetivo
  • contexto
  • dados disponíveis
  • restrições
  • formato de saída
  • critério de qualidade

Evite

  • “faça uma análise”
  • “melhore isso”
  • “seja criativo”
  • anexar tudo sem explicar
  • pedir conclusão sem método
  • aceitar resposta sem checar

objetivo

Receita de prompt para aula e trabalho

Você é [papel].

Contexto: [quem sou eu, público, situação, objetivo].

Dados disponíveis: [cole, anexe ou descreva].

Tarefa: [o que fazer exatamente].

Restrições: [não inventar, não usar API, manter simples, citar fontes].

Formato: [tabela, bullets, código, roteiro, email].

Antes de responder: liste dúvidas críticas ou hipóteses necessárias.

Gerenciar contexto é parte do trabalho

  • Contexto demais atrapalha tanto quanto contexto de menos
  • O modelo precisa saber o que é importante
  • Arquivos anexados não substituem instrução clara
  • Conversas longas acumulam ruído e decisões antigas

Regra simples

Quando a conversa ficar confusa, abra uma nova conversa com um resumo limpo: objetivo, decisões tomadas, arquivos importantes e próximo passo.

O que colocar no contexto

Inclua Por quê
objetivo final evita resposta genérica
público e tom muda linguagem e profundidade
dados ou trechos relevantes reduz alucinação
restrições protege tempo e escopo
exemplo de saída desejada melhora formato
critérios de validação força raciocínio checável

Modelo grande ou pequeno?

Use modelo pequeno quando… Use modelo grande quando…
tarefa é repetitiva tarefa é ambígua
erro é barato erro é caro
output é curto há muito contexto
há formato bem definido precisa raciocínio longo
volume importa qualidade marginal importa

Exemplo: classificar 500 notícias pode usar modelo barato; desenhar a metodologia da classificação merece modelo melhor.

Validação: nunca terceirize o juízo

  • Peça para separar fatos, inferências e opiniões
  • Peça fontes quando houver pesquisa externa
  • Rode o código
  • Confira fórmulas e unidades
  • Teste casos simples
  • Compare com uma referência independente

Sinal de alerta

Resposta fluente não é evidência de resposta correta.

Chat na web

O chat é o canivete suíço

  • Melhor ponto de entrada para quase todo mundo
  • Bom para escrita, estudo, brainstorming, dados pequenos e explicações
  • Funciona bem quando você consegue revisar o output
  • Piora quando a tarefa exige estado do projeto, execução ou dados privados

Comece no chat quando a tarefa cabe em uma conversa.

Principais opções de chat

Chat com arquivos

  • Diga o que o arquivo contém
  • Diga o que você quer extrair
  • Peça para citar página, seção, linha ou trecho
  • Peça uma tabela de achados e limitações
  • Não envie dado sensível sem política clara

Analise o PDF anexado. Quero apenas: 1. tese central; 2. três evidências; 3. duas limitações; 4. perguntas que eu deveria fazer ao autor. Se uma resposta depender de inferência, marque como inferência.

Bom uso em Finanças

  • resumir ata do COPOM ou release de resultados
  • explicar uma tabela de valuation
  • revisar premissas de um orçamento
  • transformar notas soltas em checklist
  • gerar primeira versão de código simples

Não usar sozinho para recomendação de investimento sem dados, método e revisão.

Copilot vs Cursor

Duas formas de IA para código

Copilot

Ajuda dentro do editor: autocompletar, explicar, sugerir testes e responder perguntas sobre arquivos.

Cursor

Editor pensado para IA: contexto do projeto, chat, regras, edição multiarquivo e agentes.

Chat web

Bom para rascunhar funções pequenas, mas longe do estado real do projeto.

Copilot vs Cursor: comparação prática

Dimensão GitHub Copilot Cursor
Onde vive VS Code, JetBrains, GitHub, CLI editor próprio baseado em VS Code
Melhor uso completar código e ajudar no fluxo existente mudanças com muito contexto do projeto
Curva de adoção baixa se você já usa IDE comum média, exige mudar editor ou workflow
Agentes Copilot agent mode e cloud agent Agent, rules, background/cloud agents
Risco aceitar sugestão sem entender deixar agente mexer em escopo grande

Microdemo 3: função pequena no editor

Estou em um arquivo Python. Crie uma função chamada resumir_retornos.

Input: DataFrame com colunas date, ticker, price.

Output: DataFrame com uma linha por ticker e colunas: - retorno acumulado; - volatilidade diária; - primeira data; - última data.

Use pandas, mantenha o código simples, trate preços faltantes e adicione um exemplo mínimo de uso.

Depois da sugestão: rodar, ler erro, pedir correção com a mensagem exata.

Como usar copilots sem virar passageiro

  • Escreva o nome da função antes de pedir ajuda
  • Defina inputs e outputs
  • Peça uma solução simples antes da elegante
  • Aceite trechos pequenos
  • Leia o diff
  • Rode um teste mínimo

Regra

Autocomplete acelera digitação. Não substitui entendimento.

Ferramentas especializadas

Por que usar ferramentas especializadas?

  • Chat geral tenta resolver tudo
  • Ferramenta especializada limita o problema
  • Menos liberdade pode significar mais confiabilidade
  • O ganho aparece quando há fontes, documentos ou fluxo repetido

NotebookLM

  • Melhor quando você tem documentos próprios
  • A unidade de trabalho é o conjunto de fontes
  • Bom para estudar, preparar aula, revisar relatórios e comparar documentos
  • Ajuda a manter respostas ancoradas no material

Uso ideal

Suba poucos documentos bons, faça perguntas específicas e peça evidências do próprio material.

Terminal mínimo

Por que falar de terminal?

  • Agentes usam terminal para executar trabalho
  • Erros aparecem como texto
  • Você precisa saber onde está e o que mudou
  • Poucos comandos já reduzem muito a dependência

Não é virar programador avançado. É conseguir acompanhar a máquina trabalhando.

Comandos que bastam para começar

Comando Pergunta que responde
pwd em que pasta estou?
ls quais arquivos existem aqui?
cd pasta como entro em outra pasta?
python script.py o script roda?
quarto render slides_dia2.qmd o deck renderiza?
git status o que mudou?
git diff quais linhas mudaram?

Ler erro sem pânico

Procure

  • última linha do erro
  • arquivo e número da linha
  • nome da exceção
  • variável ou pacote faltante

Peça ajuda assim

Explique este erro e proponha a menor correção.
Não reescreva o projeto inteiro.

[cole o comando executado]
[cole o traceback completo]

Agentes

O que muda com agentes?

  • Chat responde
  • Copilot sugere
  • Agente executa uma tarefa com ferramentas

Um agente pode ler arquivos, editar código, rodar testes, abrir navegador, buscar contexto e iterar.

Fluxo de trabalho com agentes

1. Planejar

escopo, arquivos, riscos

2. Editar

mudança pequena e rastreável

3. Testar

comando real e erro real

4. Revisar

diff, resultado e limites

Exemplos de agentes

Ferramenta Onde trabalha Bom para
Claude Code terminal e projeto local código, scripts, refatoração, debugging
Codex workspace, terminal, browser, repositório implementação, testes, PRs, tarefas longas
GitHub Copilot agent IDE/GitHub issues, mudanças em projeto, PRs
Cursor Agent editor Cursor edição multiarquivo com contexto do workspace

Microdemo 6: pedido para agente

No projeto atual, crie um slide novo no deck do Dia 2 explicando quando usar APIs.

Regras: - leia primeiro _quarto.yml, styles.css e o deck existente; - mantenha o estilo visual atual; - não edite arquivos gerados; - depois rode conda run -n FURG quarto render dia2/slides_dia2.qmd; - no final, diga quais arquivos mudou e qual comando de teste rodou.

Compare isso com “faz um slide sobre APIs”. A diferença é escopo, contexto e verificação.

Dar autonomia com limites

  • Defina arquivos que podem ser editados
  • Diga o que não deve mudar
  • Peça plano antes de mudança grande
  • Peça teste depois da edição
  • Peça resumo do diff
  • Revise antes de commitar

Cuidado

Quanto mais amplo o pedido, maior o risco de uma solução bonita resolver o problema errado.

Quando agentes atrapalham

  • tarefa ainda não está clara
  • não há teste ou critério de aceite
  • dados são sensíveis
  • mudança é política, não técnica
  • você não consegue revisar o resultado

Agente bom acelera trabalho bem definido. Ele não salva especificação ruim.

APIs

Quando usar API

  • tarefa repetida muitas vezes
  • precisa integrar em sistema, planilha ou app
  • precisa controlar formato e logs
  • precisa rodar em lote
  • precisa comparar custo por volume

Se é uma tarefa única, chat ou agente costuma ser melhor.

Como pensar custo de API

Componente O que significa
input tokens tudo que você envia: prompt, contexto, arquivos resumidos
output tokens tudo que o modelo responde
modelo pequeno, grande, reasoning, multimodal
ferramentas busca web, execução de código, imagens, áudio
cache/batch modos para reduzir custo em escala

O maior erro é mandar contexto demais para uma tarefa simples.

Microdemo 7: API sem chave real

from openai import OpenAI

client = OpenAI(api_key="NAO_COLE_CHAVE_NO_SLIDE")

resposta = client.responses.create(
    model="modelo-pequeno",
    instructions="Classifique em positivo, neutro ou negativo.",
    input="A empresa reportou lucro acima do esperado.",
)

print(resposta.output_text)

Exemplo conceitual. Em aula, não usar chave real em slide, GitHub ou gravação.

Boas práticas com API

  • chave em variável de ambiente, nunca no código
  • limite de gasto configurado
  • logs de entrada e saída
  • modelos pequenos para tarefas simples
  • fallback quando API falhar
  • teste com amostra pequena antes de rodar em lote

API em Finanças: bons exemplos

  • classificar notícias por tema
  • resumir relatórios de resultados
  • extrair campos de documentos padronizados
  • gerar primeira versão de comentário mensal
  • criar assistente interno com base em políticas e documentos

Evitar: decisão financeira automática sem validação, governança e responsabilidade clara.
